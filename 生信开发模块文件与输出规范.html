📋 生信开发模块文件与输出规范

本规范适用于生信分析模块开发（如 RawData 处理、差异分析、富集分析等），用于统一脚本组织、参数记录、结果追溯与报告产出。

目录说明

📂 scripts/ - 内部实现脚本目录

  • 顶层入口固定：模块使用者统一执行顶层 run.sh，不直接面向 scripts/ 内部脚本
  • 存放该模块的内部实现脚本（.R、.py、.sh 等），可由 scripts.sh 或其他调度脚本串联执行
  • 内部实现可调整：scripts/ 内部脚本后续可做保护处理（如加密、编译、二进制化等）、替换或重构，但应保持 run.sh 的对外调用方式稳定
  • 路径处理统一：所有输入、输出、配置、日志路径在使用前都应先做 normalizePath 或等价标准化处理，禁止直接拼接原始路径字符串后立即读写文件
  • 计算与作图分离：同一分析点的计算逻辑和可视化逻辑应拆分为独立脚本，如 DEGs_analysis.R（差异计算）与 DEGs_plot.R（热图/火山图等），便于只改图时不必重跑计算、提高复用性和调试效率
  • 脚本命名使用驼峰命名法，清晰表明功能，建议按分析流程顺序命名
  • 配置模板：在 config/ 下保存示例配置（*.config.example.ini/yaml）
  • 模板与运行配置分离：config/ 保存模板，result/分析点/ 保存运行时配置并可覆盖
  • 读取文件推荐：R脚本推荐使用 data.table 包
  • 示例：
    • run.sh - 模块对外执行入口
    • scripts/RawData.R - 原始数据处理内部脚本
    • scripts/DEGs_analysis.R - 差异表达计算脚本
    • scripts/DEGs_plot.R - 差异表达可视化脚本
    • scripts/scripts.sh - 内部流程调度脚本
    • config/DEGs.config.example.yaml - 参数模板
💡 计算与作图分离 — 实施约定
  • 中间结果格式：计算脚本将分析结果保存为 .qs2（R）或 .csv（通用），作图脚本直接读取中间结果文件，不重复执行计算逻辑
  • 中间结果位置：中间结果统一输出到 result/分析点/，与最终结果同目录，命名建议加 _intermediate 后缀或放入 _tmp/ 子目录以便区分
  • 命名规范：计算脚本以 _analysis 结尾，作图脚本以 _plot 结尾，如 DEGs_analysis.R / DEGs_plot.R、GSEA_analysis.R / GSEA_plot.R
  • 调度串联：scripts.sh 或 run.sh 中先调用计算脚本再调用作图脚本，支持通过参数单独运行作图脚本而不触发计算（如 run.sh --plot-only）
  • 参数传递：作图脚本的可视化参数（色板、字号、图幅尺寸等）应独立于分析参数，可在配置文件中设专用 [Plot] 段
  • 日志独立：如计算和作图分开运行，各自的日志应能独立追溯；若串联运行，可合并到同一份日志中按步骤分段记录
📝 计算脚本示例（DEGs_analysis.R）： 
# DEGs_analysis.R — 差异表达计算脚本
# 输入：表达矩阵 + 分组信息
# 输出：中间结果 .qs2 + 最终 CSV

library(data.table)
library(DESeq2)
library(qs2)

# ---- 路径标准化 ----
input_expr   <- normalizePath(input_expr, winslash = "/", mustWork = FALSE)
input_group  <- normalizePath(input_group, winslash = "/", mustWork = FALSE)
output_dir   <- normalizePath(output_dir, winslash = "/", mustWork = FALSE)

# ---- 读取数据 ----
expr_mat  <- fread(input_expr, data.table = FALSE)
group_info <- fread(input_group, data.table = FALSE)

# ---- DESeq2 分析 ----
dds <- DESeqDataSetFromMatrix(countData = expr_mat, colData = group_info, design = ~group)
dds <- DESeq(dds)
res <- results(dds, alpha = p_cutoff)

# ---- 保存中间结果（供作图脚本直接读取） ----
qs2::qs_save(res, file.path(output_dir, "DEGs_intermediate.qs2"))

# ---- 保存最终 CSV ----
fwrite(as.data.frame(res), file.path(output_dir, "01.DEGs.csv"))

⚙️ 调度入口示例（run.sh 片段）： 
#!/usr/bin/env bash
# run.sh — 支持 --plot-only 跳过计算

PLOT_ONLY=false
for arg in "$@"; do
    case "$arg" in --plot-only) PLOT_ONLY=true ;; esac
done

if [ "$PLOT_ONLY" = false ]; then
    echo "[$(date)] 开始计算..."
    Rscript scripts/DEGs_analysis.R --config config/DEGs.config.yaml
fi

echo "[$(date)] 开始作图..."
Rscript scripts/DEGs_plot.R --config config/DEGs.config.yaml
💡 为什么推荐 data.table？
  • 高性能：比 base R 快得多，处理大数据集更高效
  • 内存友好：内存占用更少，可处理超过内存大小的数据集
  • 语法简洁：语法一致性好，学习曲线平缓
  • 功能丰富：类似 SQL 的语法，操作直观
  • 并行处理：支持多线程并行计算

📄 logs/ - 日志文件夹 核心

  • 存放各分析点或模块入口的运行日志
  • 日志文件命名格式：分析点名.YYYYMMDD_HHMMSS.log
  • 时间戳精确到秒：格式为 YYYYMMDD_HHMMSS（如 20260306_143022），即可同时满足唯一标识与自然排序
  • 快速定位：无需打开文件即可知道分析运行时间
  • 支持中文：日志内容可以使用中文，便于理解和阅读

📊 result/ - 结果文件夹

  • 按分析点创建子文件夹（如 RawData/, DEGs/）
  • 每个分析点文件夹内存放该分析的所有结果文件
  • 结果文件使用数字前缀排序（01., 02., 03.）
  • 包含隐藏文件 .YYYYMMDD_HHMMSS 用于追踪对应的日志
  • 配置文件：使用 分析点.config.ini 或 分析点.config.yaml 格式
  • 配置文件会被每次运行覆盖，记录最新参数
  • 图表文件：每个图表必须同时保存 .png 和 .pdf 格式
  • 示例：
    • 01.DEGs.csv - 差异基因列表
    • 02.heatmap.png + 02.heatmap.pdf - 热图（双格式）
    • DEGs.config.ini - 配置文件
    • .20260306_161545 - 追踪文件

📝 report/ - 报告文件夹

  • 按分析点创建子文件夹
  • 每个分析点的报告以 YYYYMMDD_HHMMSS.docx 格式命名
  • 报告文件名与日志文件名的时间戳对应
  • 便于追溯某次分析的具体结果和参数
  • 查找最新版本：文件按时间排序，最后一个即为最新
  • 默认格式：报告默认使用 .docx 格式
  • 推荐工具：推荐使用 Quarto 生成报告
  • 示例：
    • 20260306_143022.docx - 14:30的分析报告（默认）
    • 20260306_161545.docx - 16:15的分析报告
💡 为什么推荐 Quarto？
  • 代码与报告一体：代码和报告在同一个文档中，便于同步更新
  • 多语言支持：支持 Python、R、Julia 等多种语言
  • 多种输出格式：可输出 HTML、PDF、Word、PPT；默认输出 .docx
  • 可复现性：包含完整代码，确保分析可复现
  • 版本控制：Markdown格式便于 Git 管理

🔗 结果追踪机制

✨ 核心设计： 程序开始时生成时间戳，日志和报告使用时间戳命名，结果文件夹使用隐藏文件标识时间戳，配置文件保持简洁。

🔄 时间戳生成和使用流程

  1. 程序启动：生成时间戳 YYYYMMDD_HHMMSS（精确到秒）
  2. 记录日志：由 run.sh 驱动分析并创建 分析点名.时间戳.log，在日志开头记录该时间戳
  3. 保存结果：在结果文件夹创建 .时间戳 隐藏文件，同时保存图表的 .png 和 .pdf 格式
  4. 更新配置（可选）：覆盖 分析点.config.ini 或 分析点.config.yaml
  5. 生成报告：创建 时间戳.docx 报告文件（默认使用 Quarto 生成）

🔗 追溯关系

  • 结果 → 日志：通过隐藏文件 .时间戳 找到对应的日志文件
  • 结果 → 参数：通过日志文件（首选）或配置文件查看分析参数
  • 日志 → 报告：通过时间戳匹配找到对应的分析报告
  • 完整链条：run.sh → 内部脚本/配置 → 日志 → 结果（.png+.pdf） → 报告
  • 查找最新版本：在 logs 文件夹按名称排序，最后一个即为最新运行

🛣️ 路径规范与 normalizePath 要求

⚠️ 核心原则： 凡是进入分析流程的文件路径、目录路径、配置路径、日志路径和输出路径，都必须先经过 normalizePath 或等价标准化处理。 目标是避免 Windows/Linux/macOS 下因分隔符、相对路径、冗余层级或软链接差异导致的读取失败、重复输出和日志不可追溯。

📏 标准化规则

规范项要求说明
统一预处理路径在存在性校验、读写、复制、移动、日志记录前必须先标准化不得一部分用原始路径、一部分用标准化路径
消除冗余层级自动折叠 .、..、重复分隔符等冗余片段避免相同文件被当成不同路径
分隔符一致内部处理推荐统一为标准化后的系统合法路径；日志展示建议统一为可读性更高的单一风格避免同一日志中同时出现 \\ 与 / 混用
绝对路径优先进入核心分析步骤前，关键输入输出路径应转换为绝对路径或可唯一定位的规范路径便于跨目录调用和问题追踪
软链接与真实路径如语言/平台支持，应尽量解析到真实路径并在日志中记录避免链接路径与真实路径混淆
日志一致性日志、配置、报告中引用的同一文件路径应保持一致禁止日志写相对路径、报告写另一套绝对路径

🧪 必须使用 normalizePath 的场景

  • 读取输入文件：如 group.csv、expression.csv、gene.csv、survival.csv、参考注释文件等
  • 写出结果文件：如 result/、report/、logs/ 下的所有输出
  • 读取配置参数：配置文件中的路径字段读取后必须立即标准化，不能原样透传到后续流程
  • 跨语言调用：run.sh 调用 R/Python/其他脚本时，传递前先做标准化，避免脚本间路径口径不一致
  • 文件关联追溯：隐藏追踪文件、日志文件、报告文件之间的路径引用必须基于同一套标准化结果
  • 存在性与权限校验：先 normalizePath，后 file.exists/access，避免校验对象与实际读写对象不是同一路径
补充要求： 用户输入路径、配置路径、命令行参数路径都视为不可信原始值，必须先标准化再参与拼接、判断和输出。

💻 推荐实现方式

R 示例： 
input_file <- normalizePath(input_file, winslash = "/", mustWork = FALSE)
output_dir <- normalizePath(output_dir, winslash = "/", mustWork = FALSE)

if (!file.exists(input_file)) {
    stop("输入文件不存在: ", input_file)
}
Python 示例： 
from pathlib import Path

input_file = Path(input_file).expanduser().resolve()
output_dir = Path(output_dir).expanduser().resolve()

if not input_file.exists():
    raise FileNotFoundError(f"输入文件不存在: {input_file}")
Shell 示例： 
INPUT_FILE="$(python -c 'from pathlib import Path; import sys; print(Path(sys.argv[1]).expanduser().resolve())' "$INPUT_FILE")"
OUTPUT_DIR="$(python -c 'from pathlib import Path; import sys; print(Path(sys.argv[1]).expanduser().resolve())' "$OUTPUT_DIR")"
落地建议： 每个模块应封装统一的路径标准化函数或公共工具，不要在多个脚本中各自手写不同口径的路径处理逻辑。

🛠️ 脚本开发规范

⚠️ 核心原则： 脚本必须具备清晰的错误处理、一致的命名风格和可维护的编码规范。 不允许"静默失败"——任何异常都应被捕获、记录并以非零退出码终止。

🚨 错误处理与退出码规范

规范项要求说明
退出码约定0 = 成功；1 = 一般错误；2 = 输入校验失败；3 = 依赖/环境异常上游调度脚本根据退出码判断是否继续
Shell 错误传播set -euo pipefail 必须写在 run.sh 和所有 .sh 脚本头部任何命令失败立即终止，未定义变量报错，管道中任一环节失败即失败
R 错误捕获关键分析步骤使用 tryCatch 包裹，失败时写日志并 quit(status = 1)禁止裸 stop() 后不做日志记录
Python 错误捕获入口函数使用 try/except，异常写入日志后 sys.exit(1)禁止 except: pass 或静默吞异常
日志联动错误信息必须同时输出到 stderr 和日志文件便于交互式调试和离线追溯
清理机制Shell 使用 trap cleanup EXIT；R/Python 使用 on.exit / finally确保异常退出时临时文件被清理、日志被刷盘
R 错误处理示例： 
tryCatch({
    dds <- DESeq(dds)
}, error = function(e) {
    log_error(paste0("DESeq2 分析失败: ", conditionMessage(e)))
    quit(status = 1)
})
Python 错误处理示例： 
try:
    result = run_analysis(config)
except Exception as e:
    logger.error(f"分析失败: {e}")
    sys.exit(1)

🐚 Shell 脚本硬性规范

规范项要求示例
Shebang必须使用 #!/usr/bin/env bash确保跨环境可移植
安全模式第二行固定 set -euo pipefail任何错误立即终止
参数解析使用 getopts 或 case 循环解析命令行参数--config, --plot-only, --help
帮助信息必须实现 --help / -h，输出用途、参数说明和示例run.sh --help

run.sh 模板： 
#!/usr/bin/env bash
set -euo pipefail

# ---- 帮助信息 ----
usage() {
    cat <<EOF
用法: $(basename "$0") [选项]

选项:
  --config FILE   指定配置文件（默认: config/DEGs.config.yaml）
  --plot-only     仅运行作图脚本，跳过计算
  -h, --help      显示此帮助信息

示例:
  $(basename "$0") --config config/DEGs.config.yaml
  $(basename "$0") --plot-only
EOF
    exit 0
}

# ---- 默认参数 ----
CONFIG="config/DEGs.config.yaml"
PLOT_ONLY=false

# ---- 参数解析 ----
while [[ $# -gt 0 ]]; do
    case "$1" in
        --config)    CONFIG="$2"; shift 2 ;;
        --plot-only) PLOT_ONLY=true; shift ;;
        -h|--help)   usage ;;
        *)           echo "未知参数: $1" >&2; exit 1 ;;
    esac
done

# ---- 时间戳与日志 ----
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
LOG_FILE="logs/DEGs.${TIMESTAMP}.log"
mkdir -p logs
exec > >(tee -a "$LOG_FILE") 2>&1

# ---- 清理函数 ----
cleanup() { echo "[$(date)] 脚本退出"; }
trap cleanup EXIT

# ---- 执行分析 ----
if [ "$PLOT_ONLY" = false ]; then
    echo "[$(date)] 开始计算..."
    Rscript scripts/DEGs_analysis.R --config "$CONFIG"
fi

echo "[$(date)] 开始作图..."
Rscript scripts/DEGs_plot.R --config "$CONFIG"

echo "[$(date)] 分析完成"

✅ 推荐做法

  • 统一时间戳：程序开始时生成时间戳，所有输出文件统一使用
  • 日志优先：无论是否有配置文件，日志中必须详细记录所有参数
  • 配置文件简洁：使用 分析点.config.ini 或 分析点.config.yaml，每次运行覆盖
  • 日志使用中文：日志内容可以使用中文，便于理解和阅读
  • 计算与作图分离：同一分析点的计算脚本和作图脚本拆成独立文件，改图时无需重跑计算
  • 双格式图表：每个图表必须同时保存 .png 和 .pdf 格式（特殊情况除外）
  • 统一路径标准化：所有路径在校验、读写、记录前先做 normalizePath 或等价处理，并在日志中保持一致口径
  • 图表字体统一：图片中的英文统一使用 Arial
  • 色板标准：推荐使用 ggsci::npg、RColorBrewer 等学术配色方案；图表必须对色盲友好，避免仅靠红/绿区分数据
  • 使用 .qs2：R数据使用 .qs2 格式保存（qs2 R 包，替代 .rds/.RData）
  • 使用 data.table：读取文件推荐使用 data.table 包
  • 使用 Quarto：生成报告推荐使用 Quarto，默认输出 .docx 格式
  • 错误即终止：Shell 脚本固定 set -euo pipefail；R/Python 必须 tryCatch/try-except 并以非零退出码终止
  • 输出自检：脚本结束前自动校验输出文件存在性、非空、关键列完整
  • README.md 完备：每个模块必须包含标准 README（用途、依赖、输入输出、运行示例）
  • CHANGELOG.md：语义化版本号 + 每次变更记录
  • 冒烟测试：提供示例数据和 tests/smoke_test.sh，一键验证核心流程
  • 定期清理过期的日志文件和临时结果（默认保留最近 30 天）
  • 使用版本控制系统（Git）管理scripts文件夹中的脚本
  • 为重要分析结果创建README.md说明文件内容
  • 定期备份result和report文件夹到云端或外部存储
  • 日志文件使用结构化格式，便于后续检索和分析
  • 在日志开头添加摘要部分，包含关键统计信息

❌ 避免做法

  • 使用无意义的文件名（如 result1.csv, final.pdf）
  • 时间戳只精确到日期，导致同一天多次运行无法区分
  • 日志只记录运行成功/失败，不记录具体参数
  • 依赖配置文件而忽略日志的参数记录功能
  • 不保留日志文件，导致无法追溯分析过程
  • 结果文件与日志文件无法对应
  • 不同输出文件使用不同的时间戳
  • 跳过路径标准化：直接用命令行输入、配置值或字符串拼接后的路径做存在性校验和文件读写
  • 配置文件使用时间戳命名，导致文件夹碎片化
  • 计算与作图混写：把数据计算和可视化代码写在同一个脚本中，导致改图必须重跑全部计算
  • 只保存单一图表格式：只保存 .png 或只保存 .pdf
  • 图表字体混用：同一项目中混用 Arial、Times New Roman、宋体等
  • 使用 .rds/.RData：推荐使用更现代的 .qs2 格式（qs2 R 包）
  • 使用 base R：推荐使用更高效的 data.table
  • 静默吞异常：except: pass 或裸 tryCatch 不记录错误信息
  • 缺少 shebang/安全模式：Shell 脚本不写 #!/usr/bin/env bash 和 set -euo pipefail
  • 模块间内存传递：通过全局变量或共享内存传递数据，不走文件接口
  • 输出不校验：脚本执行完不检查输出文件是否完整和正确
  • 无 README / 无版本号：模块缺少文档或依赖说明，新人无法独立运行
  • 敏感数据入库：将含患者真实 ID 的临床表提交到 Git 仓库

R 输出校验示例： 
# 输出校验函数
validate_output <- function(file_path, required_cols = NULL) {
    if (!file.exists(file_path)) stop("输出文件缺失: ", file_path)
    if (file.size(file_path) == 0) stop("输出文件为空: ", file_path)

    if (grepl("\\.csv$", file_path)) {
        df <- data.table::fread(file_path, nrows = 5)
        if (!is.null(required_cols)) {
            missing <- setdiff(required_cols, names(df))
            if (length(missing) > 0) stop("缺少关键列: ", paste(missing, collapse = ", "))
        }
    }
    log_info(paste0("校验通过: ", file_path))
}

# 使用
validate_output("result/DEGs/01.DEGs.csv", c("SYMBOL", "log2FoldChange", "pvalue", "padj"))

# DEGs — 差异基因分析模块

> 版本: v1.2.0 | 更新: 2026-03-26

## 简介

基于 DESeq2 进行差异基因分析，输出差异基因列表、热图和火山图。

## 环境依赖

- R >= 4.2
- 关键包：DESeq2, data.table, ggplot2, qs2, pheatmap
- 完整依赖：`renv.lock`

## 上游模块依赖

| 上游模块 | 所需文件                           | 最低版本 |
|---------|-----------------------------------|---------|
| RawData | result/RawData/01.GSEXXX_count.csv | v1.0    |

## 快速开始

```bash
# 查看帮助
bash run.sh --help

# 完整运行（计算 + 作图）
bash run.sh --config config/DEGs.config.yaml

# 仅重新作图
bash run.sh --plot-only
```

## 输入文件

| 文件              | 格式  | 关键列            | 说明           |
|------------------|------|------------------|---------------|
| expression.csv   | CSV  | SYMBOL + 样本列   | 表达矩阵        |
| group.csv        | CSV  | sample, group    | 分组信息        |

## 输出文件

| 文件                    | 目录           | 说明           |
|------------------------|---------------|---------------|
| 01.DEGs.csv            | result/DEGs/  | 差异基因列表     |
| 02.heatmap.png/.pdf    | result/DEGs/  | Top 基因热图    |
| 03.volcano.png/.pdf    | result/DEGs/  | 火山图          |

## 参数说明

| 参数            | 默认值  | 说明               |
|----------------|--------|--------------------|
| p_value_cutoff | 0.05   | 显著性阈值           |
| log2fc_cutoff  | 1.0    | log2 倍数变化阈值     |
| method         | DESeq2 | 差异分析方法          |

规范项建议要求说明
物种与注释版本日志必须记录物种、参考基因组版本、注释版本示例：human | GRCh38 | GENCODE v46
基因ID体系明确使用 Ensembl ID 或 Gene Symbol，不混用跨步骤转换时记录映射方法和丢失数量

软件环境记录记录 Python/R 版本和关键包版本建议输出 sessionInfo() 或 pip freeze 摘要

📊 文件格式建议

📊 数据文件格式

  • 表格数据：优先使用 .csv（通用、兼容性好）
  • R数据：使用 .qs2 保存（qs2 R 包，替代 .rds/.RData）
💡 为什么使用 .qs2 而不是 .rds/.RData？
  • .qs2（qs2 R 包）是更现代的二进制格式，性能更好
  • 跨平台兼容性更好，不受R版本限制
  • 压缩比更高，文件更小
  • 读写速度更快

📈 图表文件格式

  • 必须同时保存：.png（位图）+ .pdf（矢量图）
  • 字体要求：使用默认字体（sans-serif 无衬线族）即可；禁止使用 Times New Roman 等衬线字体
  • PNG：300 dpi；R 中推荐使用 png(type = "cairo") 保证无头服务器兼容
  • PDF：矢量格式，可编辑、可缩放；R 中推荐使用 cairo_pdf() 或 ggsave(device = cairo_pdf)，确保字体嵌入和中文支持
  • 默认尺寸：单面板图 6×4 in，多面板按比例缩放；发表级图表建议 7×5 in
  • 特殊情况：交互图表使用 .html（Plotly等）
⚠️ 必须同时保存 PNG 和 PDF

除非是交互式图表（如 Plotly），每个静态图表都应同时保存 .png 和 .pdf。

PNG 适合演示与办公文档，PDF 适合发表与后续编辑。

📝 报告文件格式

  • 默认格式：使用 .docx（Word）
  • 其他格式：
    • .pdf（最终版本，不可编辑）
    • .pptx（演示文稿）
    • .html（网页版本）
  • 推荐工具：使用 Quarto 生成报告
💡 Quarto 补充说明（避免重复）
  • 核心优势见上文 report/ - 为什么推荐 Quarto
  • 补充：支持跨平台（Windows、Mac、Linux）与团队协作流程
📝 报告标准章节结构

无论使用 Quarto 还是其他工具，分析报告应包含以下标准章节：

  1. 分析概述 — 一句话说明本次分析的目的和方法
  2. 数据来源 — 数据集名称、样本量、来源平台（如 GEO、TCGA）
  3. 分析参数 — 所有关键参数（阈值、方法、随机种子等）
  4. 结果摘要 — 关键统计数字（差异基因数、富集通路数等）
  5. 图表展示 — 每个图表配简要解读文字
  6. 结论与讨论 — 结果解读、局限性说明
  7. 软件与版本 — R/Python 版本、关键包版本（可附 sessionInfo 输出）
  8. 附录（可选） — 补充表格、完整参数列表、辅助图表

🔍 输入文件与结构化校验规范

📁 输入文件清单（CSV）

文件名（推荐）关键要求说明
group.csv两列：sample、groupgroup 内容可变，但字段名固定；每组建议至少 3 个样本
expression.csv行为基因，列为样本，首列必须为 SYMBOLSYMBOL 必须大写；列名（除 SYMBOL 外）必须与 group.csv 的 sample 列完全一致；需在日志中注明数据类型（如原始计数 counts / TPM / FPKM）
gene.csv列名必须为 SYMBOL基因列表输入
riskscore.csv需包含样本与风险评分字段风险分值必须为数值型
survival.csv必须包含 OS 与 OS.timeOS 仅允许 0/1；OS.time 必须 > 0；如有其他终点，字段命名统一为 <终点> + <终点>.time（如 PFS/PFS.time、DFS/DFS.time）
命名补充： 允许扩展命名（如 gene_v1_human.csv），但推荐优先使用标准文件名以减少解析歧义。

✅ 字段类型与取值约束

字段类型要求取值/约束
sample字符型（chr）不能为空，建议全局唯一
group字符型（chr）允许业务自定义分组标签
OS整数型（int）仅允许 0 或 1
OS.time数值或整数型（num/int）必须 > 0
RS / 风险评分列数值型（num）允许负值，不允许 NA
riskgroup字符型（chr）仅允许 High 或 Low（注意首字母均大写）
SYMBOL字符型（chr）列名必须大写 SYMBOL
硬性规则： 全部输入文件必须为 .csv；文件编码必须为 UTF-8（推荐无 BOM）；行尾符建议统一为 LF（\n）；数据中不允许出现 NA；数值字段必须可解析为 num/int。
