🛡️ 生信模块代码保护指南

本指南用于约束模块对外交付时的源码暴露边界、入口脚本设计、配置模板留存策略以及保护实现原则，确保模块可用、可维护、可保护。

主入口固定为 run.sh 内部实现集中在 scripts/ 保护实现不改变外部调用方式

🎯 核心原则

  1. 对外只暴露稳定入口：使用者统一通过 run.sh 调用模块，不应直接依赖 scripts/ 内部脚本路径。
  2. 内部实现与外部接口解耦：即使后续对 scripts/ 做保护处理（如加密、编译、二进制化等），run.sh 的参数约定也应保持稳定。
  3. 可公开的只有必要内容：保留运行所需说明、参数模板、结果结构说明；不公开不必要的中间脚本、调试脚本和历史实验代码。

📁 推荐目录边界

module_name/
├── run.sh
├── scripts/
│   ├── scripts.sh
│   ├── RawData.R
│   └── DEGs.py
├── config/
│   └── DEGs.config.example.ini/yaml
├── logs/
├── result/
├── report/
├── README.md           # 必备（参见《规范》README 必备内容章节）
├── tests/              # 冒烟/回归测试（参见《规范》质量保证章节）
├── CHANGELOG.md        # 版本变更记录
├── cleanup.sh          # 临时文件清理脚本
├── requirements.txt    # Python 依赖锁定
└── renv.lock           # R 依赖锁定（二选一或同时保有）
推荐做法： 对外文档只讲清楚 run.sh 的用法、输入、输出和错误定位方式；scripts/ 仅作为内部实现目录维护。

🔍 哪些内容可以公开，哪些应受保护

内容 建议状态 说明
run.sh 公开 这是模块稳定入口，应明确参数、帮助信息、退出码和日志位置。
README.md 公开 记录运行方式、输入输出说明、依赖要求，属于外部使用所必需（参见《规范》README 必备内容章节）。
config/*.example.ini/yaml 公开 参数模板应保留，便于使用者理解输入要求和默认值。
scripts/*.R、scripts/*.py 可保护 属于内部实现，可根据交付策略做保护处理（如加密、编译、二进制化、打包等）或受控发布。
临时调试脚本、试验代码 禁止交付 这类文件不稳定、不可追溯，也容易暴露无关实现细节。
历史结果、缓存、中间文件 禁止交付 交付包中只保留必要模板与正式输出结构，不夹带开发过程残留。
requirements.txt / renv.lock 公开 环境依赖锁定文件属于可复现运行的必要条件，应随交付包一并提供。
CHANGELOG.md 公开 版本变更记录便于使用者了解各版本差异，推荐保留。

🔐 run.sh 的保护要求

  • 必须提供清晰的参数入口，优先支持配置文件路径、输入数据路径、输出目录等核心参数。
  • 必须输出可定位问题的日志路径，不把错误完全吞掉。
  • 内部脚本名可以变化，但 run.sh 的调用方式不能随保护策略频繁变化。
  • 如果内部脚本经过保护处理（如加密、编译等），run.sh 仍应保持原有帮助信息和返回行为。
不推荐： 让使用者直接运行 python scripts/DEGs.py 或 Rscript scripts/RawData.R。这会让外部调用与内部实现强绑定，后续一旦做保护处理就会破坏兼容性。

⚙️ run.sh 参数与退出码规范

接口项 建议要求 说明
帮助信息 必须支持 -h 或 --help 帮助信息至少说明用途、必填参数、可选参数、示例命令和输出位置。
配置入口 优先支持 -c 或 --config 配置文件路径应是最稳定的接口之一，便于保护内部实现时保持兼容。
输入输出参数 建议显式支持输入、输出目录、线程数等关键参数 不要让使用者依赖内部默认路径或脚本内硬编码路径。
日志输出 必须打印日志文件位置 运行开始、失败退出和成功结束时都应能快速定位日志。
退出码 成功返回 0，失败返回非 0 不同类型错误可进一步细分退出码，但至少保证成功与失败可被外部脚本识别。 
示例：
./run.sh --config config/DEGs.config.example.ini --input ./input --output ./result

推荐退出码约定：
0   运行成功
1   参数错误
2   输入文件缺失或格式错误
3   运行环境异常
4   内部分析流程执行失败
接口稳定原则： 就算内部从明文脚本切到受保护形式（如加密、编译、二进制化等），run.sh 的主要参数名称和退出码语义也不要轻易变化。

🔒 scripts/ 的保护要求

  • scripts/ 视为内部实现目录，可按需要做保护处理（如加密、编译、二进制化、压缩等）或限制访问。
  • 保护后仍需保留稳定的日志输出、结果目录结构和配置映射关系。
  • 若使用中间调度脚本，如 scripts.sh，应由 run.sh 统一调用，不单独对外宣讲。
  • 调试用开关、测试参数和临时路径不应在对外交付版本中保留。
  • 保护后路径可能发生变化，仍须遵循 normalizePath() 规范（详见《生信开发模块文件与输出规范》路径规范章节），确保跨平台兼容。
禁止事项： 不要把密钥、临时账号、内网路径、调试数据源、测试样本或历史分析结果一并打包到交付目录中。

📊 保护策略分级建议

级别 适用场景 建议做法
基础保护 内部协作、受控团队共享 保留源码目录结构，但移除调试脚本、测试数据和无关历史文件，只保留 run.sh、内部脚本、模板配置和必要说明。
中等保护 外部合作、需要限制实现细节暴露 对 scripts/ 中核心脚本做保护处理，保留明文 run.sh 与配置模板，并确保日志与错误信息可读。
工具参考：R — compile() 字节码编译；Python — pyinstaller、cython 编译为二进制。
强化保护 正式交付、商业化或需严格控制源码访问 仅公开稳定入口、模板配置和结果约定；内部脚本使用受保护方式或受控分发，并配套交付检查记录。
工具参考：R — encryptedRmd 加密、Docker 镜像封装；Python — nuitka 全编译、pyarmor 加密混淆。
选择原则： 保护级别越高，越要重视外部兼容性。无论采用哪种保护方式，都不应让使用者承担额外的内部实现知识。

🛠️ 具体保护实施方法

1. 文件权限控制（基础保护 — 所有语言通用）

# 目录：仅所有者和同组可进入
chmod 770 scripts/

# 脚本文件：所有者可读写执行，同组可读执行，其他人无权限
chmod 750 scripts/*.R scripts/*.py scripts/*.sh

# run.sh 保持所有人可执行（对外入口）
chmod 755 run.sh

# 配置模板和 README：所有人可读
chmod 644 config/*.example.* README.md CHANGELOG.md
说明： chmod 770 让 scripts/ 目录仅所有者和同组用户可访问；chmod 750 让脚本对组外用户不可读。适合内部团队协作场景，Linux/macOS 下有效。

2. R 脚本保护

方法命令 / 步骤保护强度注意事项
字节码编译 
library(compiler)
cmpfile("scripts/DEGs_analysis.R",
        "scripts/DEGs_analysis.Rc")
# 运行：loadcmp("scripts/DEGs_analysis.Rc")
 中等 可加速执行；.Rc 文件不可直接阅读，但有反编译风险。
打包为 R 包 
# 1. 创建包骨架
usethis::create_package("MyModule")
# 2. 将核心函数放入 R/ 目录
# 3. 编译安装（生成二进制包）
R CMD INSTALL --build MyModule
# 4. run.sh 中调用：
#    Rscript -e "library(MyModule); main()"
 中-强 编译安装后源码不在包内；run.sh 调用方式不变。
Docker 镜像封装 
# Dockerfile（R 环境）
FROM rocker/r-ver:4.3.2
COPY scripts/ /app/scripts/
COPY run.sh /app/
WORKDIR /app
ENTRYPOINT ["bash", "run.sh"]
# 构建：docker build -t mymodule:v1 .
# 运行：docker run -v $(pwd)/input:/data mymodule:v1 \
#        --config /data/config.yaml
 强 源码封装在镜像层内；用户只能通过 run.sh 入口调用。

3. Python 脚本保护

方法命令 / 步骤保护强度注意事项
字节码编译 (.pyc) 
# 编译所有 .py → .pyc
python -m compileall scripts/
# 删除源码，只保留 __pycache__/*.pyc
# run.sh 中调用：python scripts/__pycache__/DEGs.cpython-311.pyc
 低 可被 uncompyle6 反编译；仅防止随意阅读。
PyInstaller 打包 
# 打包成单文件可执行
pyinstaller --onefile scripts/DEGs.py \
  -n DEGs_runner --distpath scripts/protected_runtime/
# run.sh 中调用：
#   ./scripts/protected_runtime/DEGs_runner --config ...
 中-强 生成平台相关二进制；需在目标平台编译。
Cython 编译 
# 1. 将 .py 重命名为 .pyx（或直接编译）
# 2. 创建 setup.py：
from setuptools import setup
from Cython.Build import cythonize
setup(ext_modules=cythonize("scripts/DEGs.pyx"))
# 3. 编译：python setup.py build_ext --inplace
# 生成 .so/.pyd 二进制文件
 强 编译为 C 扩展；难以逆向；需安装 Cython。
PyArmor 加密 
# 加密整个 scripts 目录
pyarmor gen -O scripts/protected_runtime/ scripts/*.py
# 运行（需 pyarmor 运行时）：
#   python scripts/protected_runtime/DEGs.py
 强 运行时解密；可设过期时间和机器绑定；商业版功能更多。

4. Shell 脚本保护

# 使用 shc（Shell Script Compiler）
# 安装：apt-get install shc  或  brew install shc
shc -f scripts/scripts.sh -o scripts/protected_runtime/scripts
# 生成二进制可执行文件；删除中间 .x.c 文件
rm -f scripts/scripts.sh.x.c

# run.sh 中改为调用编译后的二进制：
# ./scripts/protected_runtime/scripts "$@"
保护后验证清单：
  • 保护后 run.sh 仍可正常运行且参数不变
  • 日志输出完整，错误信息可读
  • 结果目录结构与保护前一致
  • 路径使用 normalizePath() 确保跨平台兼容
  • 外部用户无需安装保护工具即可使用（PyArmor 除外，需运行时）

❓ 常见问题

  • Q：配置模板要不要一起交付？
    A：要。模板配置属于外部使用接口的一部分，不应因代码保护而缺失。
  • Q：保护处理（如加密、编译等）后报错看不懂怎么办？
    A：需要在 run.sh 和日志中保留清晰的错误提示，不要把内部异常全部变成无意义的返回码。
  • Q：可以只交付内部脚本，让用户自己写入口吗？
    A：不建议。这样会把外部接口设计责任转嫁给使用者，也不利于后续兼容与保护。
  • Q：流程调度脚本要不要对外说明？
    A：通常不需要。对外只说明主入口和必要参数，内部调度细节留在保护域内即可。

🚫 交付红线清单

红线项 禁止原因 处理方式
内网地址、账号、密钥、令牌 直接构成安全风险，可能导致环境或数据被非授权访问。 交付前全部清理；如必须配置，改为模板占位符并由外部注入。
调试脚本、测试脚本、临时 notebook 内容不稳定，容易泄露实现细节，也会混淆正式入口。 从交付包移除，仅在内部仓库保留。
历史结果、中间缓存、测试样本 容易泄露业务数据或误导用户将旧结果当成正式输出。 交付前清空，只保留目录结构和必要示例模板。
要求用户直接调用内部脚本的说明 破坏对外接口稳定性，后续一旦做保护处理就会导致兼容性问题。 统一改为通过 run.sh 调用，并删除对内部路径的直接说明。
与当前版本不一致的配置模板或帮助文档 会导致使用者按旧参数运行，产生错误或结果偏差。 交付前逐项核对模板、README、日志说明与当前版本是否一致。
执行原则： 只要某项内容不属于“外部使用所必需”，默认不进入交付包；拿不准时，先放入内部保留区，再评估是否需要公开。

🗺️ 保护指南总览流程

1. 定义外部接口

先固定 run.sh 的参数、帮助信息、日志定位方式和输出约定。

2. 划分保护边界

明确哪些内容公开、哪些放在 scripts/ 内部保护区、哪些禁止交付。

3. 执行打包检查

清理调试脚本、旧结果、密钥与缓存，核对模板配置和说明文档是否与当前版本一致。

4. 验证交付可用性

以外部使用者视角运行 run.sh，确认日志、结果、报告与错误提示都能正常工作。

推荐顺序： 先设计接口，再做保护；先完成保护，再做打包；最后必须按交付视角做一次完整验证。

✅ 交付前检查清单

  • 确认对外文档只描述 run.sh 的用法，不要求用户直接调用内部脚本。
  • 确认 config/ 中保留了可读的参数模板，且与当前版本一致。
  • 确认交付包中不包含调试脚本、旧结果、测试数据和无关缓存。
  • 确认日志、结果、报告目录结构仍符合规范页要求。
  • 确认保护处理（如加密、编译、二进制化等）后，仍可通过日志追溯到分析点和运行时间。
  • 确认 requirements.txt 或 renv.lock 已包含在交付包中，且依赖版本可复现。
  • 确认路径使用 normalizePath() 或等价处理，保护后路径仍然跨平台兼容。
  • 确认 README.md 和 CHANGELOG.md 与当前交付版本一致。
交叉引用： 本检查清单与《生信开发模块文件与输出规范》中的输出校验规范、README 必备内容、版本管理细则章节配套使用。
